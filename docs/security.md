# 安全模型

ClipBridge 的目标是让服务端承担身份管理、在线路由和临时存储职责，但不能解密剪贴板正文或敏感内容元数据。

## 加密流程

每条剪贴板内容的发送流程：

1. 发送设备为本条内容生成随机 256-bit DEK。
2. 正文使用 AES-256-GCM 分块加密；协议版本、内容 ID、源设备 ID、内容类型、块序号和末块标记被绑定进 AAD。
3. 文件名、图片尺寸和富文本格式等敏感元数据使用同一 DEK 独立加密。
4. 发送设备为每台在线目标设备分别使用其 HPKE 公钥封装 DEK。
5. 服务端保存一份正文密文和多份逐设备 wrapped DEK，并只向对应目标设备投递。

固定密码套件：

| 组件 | 算法 |
| --- | --- |
| HPKE KEM | ML-KEM-768-X25519 |
| HPKE KDF | HKDF-SHA256 |
| HPKE AEAD | AES-256-GCM |
| 正文与元数据 AEAD | AES-256-GCM |
| 密文完整性摘要 | SHA-256 |

分块 AEAD 会检测错误密钥、篡改、块重排、删除、截断和尾部附加数据。大文件以流式方式加解密，不要求把完整明文载入内存。

## 信任建立

### 服务器证书

设备端口使用服务端自签名证书。首次配对时，客户端展示实际 TLS 握手取得的 SHA-256 指纹，用户需要与 Web 配对页显示的指纹核对；确认后客户端固定该指纹。

后续指纹变化会阻断连接。用户必须重新核对并显式信任，或重置客户端后重新配对。

### 设备公钥

每台客户端在本地生成 HPKE 密钥对，私钥不会上传。客户端首次与某个目标设备同步时缓存其公钥指纹；后续公钥变化会阻断对该设备的同步并持续告警。

Web 控制台和客户端「设备互验」页面可以显示设备公钥指纹。对安全要求更高的环境应在两台设备上通过独立渠道人工对照。

## 服务端可见信息

| 服务端可见 | 服务端不可见 |
| --- | --- |
| 用户与设备 ID | 剪贴板正文 |
| 设备名称、平台、在线状态 | 文件名 |
| 源设备、目标设备和投递状态 | 图片尺寸 |
| 内容类型 | 富文本格式与纯文本回退 |
| 密文大小、时间和 SHA-256 | 内容 DEK |
| 每台设备的公钥与 wrapped DEK | 设备 HPKE 私钥 |

服务端会保留同步与管理操作所需的最小日志，但不会记录 token、私钥、密码、完整配对码或明文内容。

## 临时存储

- 内容只会投递给上传时在线的同用户设备。
- 正文只加密并上传一份，逐设备独立封装 DEK。
- 所有目标设备确认或拒绝后立即删除密文。
- 未完成的密文最长保留 300 秒，由清理任务删除。

因此 ClipBridge 不提供离线队列或长期剪贴板历史。

## 威胁边界

能够防护：

- 被动读取服务端数据库、日志或密文目录。
- 服务端管理员直接查看剪贴板正文。
- 网络中间人替换已固定的设备端证书。
- 已建立 TOFU 后，服务端静默替换目标设备公钥。
- 传输或存储过程中的正文密文篡改、截断和重排。

不负责防护：

- 已失陷的发送端或接收端；内容在系统剪贴板中本来就是明文。
- 恶意服务端拒绝服务、延迟或丢弃投递。
- 首次配对时用户未核对证书指纹。
- 首次接触目标设备公钥时未进行跨设备人工互验。
- 操作系统、WebView、浏览器扩展或其他本地进程读取剪贴板。

## 部署注意事项

- `8443` 设备端口应由客户端直接连接，不要在前面做 TLS 终止，否则客户端固定的是代理证书而不是 ClipBridge 服务端证书。
- `8080` Web 端口是 HTTP，公网部署必须置于可信 HTTPS 反向代理后。
- `clipbridge.db` 和 `certificates/` 必须持久化并备份；证书目录丢失会触发所有客户端的指纹变化告警。
- 生产环境应固定 Docker 镜像版本，不依赖浮动 `latest`。
